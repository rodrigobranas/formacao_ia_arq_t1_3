# ADR-003: tsx for Backend Development Runtime

## Status

Accepted

## Date

2026-04-06

## Context

The backend workspace uses TypeScript and needs a development workflow with hot-reload. We need to choose how TypeScript is executed during development — whether to compile first and run JavaScript, or run TypeScript directly.

## Decision

Use `tsx` (esbuild-based TypeScript executor) with its built-in `--watch` mode for backend development. No separate `tsc` compilation step for development. TypeScript is executed directly by tsx at runtime.

## Alternatives Considered

### Alternative 1: tsc + node with nodemon

- **Description**: Compile TypeScript to JavaScript using `tsc`, run the output with `node`, and use `nodemon` to watch for changes and re-trigger the build.
- **Pros**: Students see the explicit compilation step, closer to production workflow, generates inspectable JavaScript output.
- **Cons**: Slower feedback loop (compile then restart), requires configuring both tsc and nodemon, more files to manage (dist/ directory, nodemon.json).
- **Why rejected**: Added complexity and slower restarts outweigh the educational benefit of seeing compiled output at this stage.

### Alternative 2: ts-node with nodemon

- **Description**: Use `ts-node` to run TypeScript directly with `nodemon` for file watching.
- **Pros**: Well-established, runs TypeScript without build step.
- **Cons**: Significantly slower than tsx (uses TypeScript compiler internally), requires separate nodemon configuration.
- **Why rejected**: Slower startup and restart times compared to tsx; tsx provides a simpler single-tool solution.

## Consequences

### Positive

- Fast development restarts thanks to esbuild-based compilation
- Single tool handles both execution and file watching (`tsx watch`)
- No `dist/` directory or build artifacts to manage during development
- Fewer configuration files (no nodemon.json, no build scripts)

### Negative

- Students don't see the TypeScript-to-JavaScript compilation step explicitly
- tsx behavior may differ subtly from tsc in edge cases (esbuild vs TypeScript compiler)

### Risks

- Minimal risk given the simple TypeScript usage in this project; no advanced TypeScript features that would cause esbuild/tsc divergence

## Implementation Notes

Backend `package.json` will have `tsx` as a dev dependency. The dev script will be `tsx watch src/index.ts`. A `tsconfig.json` is still needed for editor support and type checking.
