# ADR-002: Makefile as Primary Command Interface

## Status

Accepted

## Date

2026-04-06

## Context

The monorepo needs a way to orchestrate commands across both frontend and backend workspaces — starting dev servers, running tests, setting up the database, etc. The PRD specifies that students should be able to run both workspaces concurrently with simple commands. We need to decide where these orchestration commands live.

## Decision

Use a root-level Makefile as the primary command interface. The root `package.json` declares npm workspaces but contains no scripts. All developer-facing commands (`make dev`, `make test`, `make setup`) are Makefile targets.

## Alternatives Considered

### Alternative 1: npm scripts with concurrently package

- **Description**: Add `concurrently` as a root dev dependency and define scripts in root `package.json` that run both workspaces.
- **Pros**: Stays within the npm ecosystem, color-coded labeled output, cross-platform.
- **Cons**: Adds a dependency, npm script syntax is limited for complex orchestration, JSON escaping issues.
- **Why rejected**: Adds an unnecessary dependency and forces complex commands into JSON string values.

### Alternative 2: npm workspaces --ws flag

- **Description**: Use built-in `npm run dev --ws` to run scripts across workspaces.
- **Pros**: Zero dependencies, built into npm.
- **Cons**: Runs workspaces sequentially by default, interleaved output is hard to read, limited control over process lifecycle.
- **Why rejected**: Sequential execution doesn't work for concurrent dev servers; output readability is poor.

## Consequences

### Positive

- Makefile syntax is explicit and readable — each target clearly shows what commands run
- No additional npm dependencies for orchestration
- Makefile is a well-established tool students will encounter in professional environments
- Easy to add new targets as the project grows

### Negative

- Make is not natively available on Windows (requires WSL, Git Bash, or chocolatey install)
- Students unfamiliar with Makefiles may need a brief introduction

### Risks

- Windows compatibility; mitigated by documenting WSL as the recommended Windows development environment

## Implementation Notes

The Makefile will include targets for: `setup` (install + docker compose), `dev` (run both workspaces), `test` (run both test suites), and individual workspace targets.
