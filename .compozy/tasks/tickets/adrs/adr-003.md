# ADR-003: File Attachments as Base64 in JSON Payloads

## Status

Accepted

## Date

2026-04-07

## Context

The ticket system needs file attachment support for both ticket creation (public) and operator comments. Files must be stored as base64 in the database with a 1MB limit per file. The existing API uses JSON exclusively for all request/response payloads.

## Decision

Send file attachments as base64-encoded strings within the standard JSON request body. The client reads files using the FileReader API, converts to base64, and includes them in the JSON payload alongside other fields. The server validates the base64 string size (max ~1.37MB encoded, representing ~1MB decoded) and stores it directly in a `ticket_attachments` table.

## Alternatives Considered

### Alternative 1: Multipart Form Data

- **Description**: Use `multipart/form-data` with a middleware like `multer` to handle file uploads
- **Pros**: Industry standard for file uploads; more efficient for large files; native browser support via FormData
- **Cons**: Requires new middleware dependency; breaks the existing JSON-only API pattern; more complex request handling; mixed content types in the same endpoint
- **Why rejected**: Adds unnecessary complexity and a new dependency. The 1MB file limit makes base64 overhead negligible. Keeping JSON-only is consistent with the existing codebase.

## Consequences

### Positive

- No new dependencies required
- Consistent with existing JSON-only API pattern
- Simpler backend — no multipart parsing, just JSON body validation
- Attachments travel with the rest of the request data atomically

### Negative

- ~33% size overhead from base64 encoding (1MB file becomes ~1.37MB in transit)
- Express JSON body parser limit must be increased to accommodate encoded files
- Not suitable for files larger than a few MB (acceptable given the 1MB limit)

### Risks

- Large base64 strings in JSON could cause memory pressure on high-traffic endpoints. Mitigation: enforce strict size validation before processing; the 1MB limit keeps payloads manageable.

## Implementation Notes

- Increase Express `express.json()` limit to `2mb` to accommodate base64-encoded 1MB files plus other fields.
- Store each attachment as a row in `ticket_attachments` with columns: `id`, `ticket_id`, `ticket_comment_id` (nullable), `filename`, `content_type`, `content` (TEXT for base64), `created_at`.
- Validate base64 size server-side before insert.
