# ADR-002: Public Endpoint Routing via Organization Slug

## Status

Accepted

## Date

2026-04-07

## Context

The ticket management system requires public-facing endpoints for ticket submission and tracking that do not require authentication. These endpoints must be scoped to a specific organization so tickets are correctly associated. The system currently identifies organizations by numeric ID internally.

## Decision

Introduce a `slug` field on the `organizations` table and use it in public endpoint URLs: `/api/public/:orgSlug/tickets`. The slug is a URL-safe, lowercase, unique identifier derived from the organization name (e.g., "acme-corp").

Public endpoints:
- `POST /api/public/:orgSlug/tickets` — submit a new ticket
- `GET /api/public/:orgSlug/tickets/:code` — track a ticket by its code

## Alternatives Considered

### Alternative 1: Organization ID in URL

- **Description**: Use `/api/public/organizations/:orgId/tickets` with the numeric ID
- **Pros**: No schema changes needed; simple implementation
- **Cons**: Exposes internal sequential IDs; enables enumeration; less user-friendly URLs
- **Why rejected**: Exposing auto-increment IDs is a security concern and makes URLs hard to share or embed.

### Alternative 2: Organization ID in Request Body/Query

- **Description**: Send organization ID in the JSON body or query string rather than the URL path
- **Pros**: No URL structure changes
- **Cons**: Not RESTful; GET requests cannot use body; harder to share links for ticket tracking
- **Why rejected**: Breaks REST conventions and makes the tracking URL unusable as a simple link.

## Consequences

### Positive

- Clean, shareable, human-readable public URLs
- Organization identity not leaked via sequential IDs
- Slug can be used in future public-facing features

### Negative

- Requires adding a `slug` column to `organizations` and generating it during signup
- Slug uniqueness must be enforced at the database level
- Existing organizations need a slug backfill (handled in init.sql since system is in development)

### Risks

- Slug collisions from similar organization names. Mitigation: append a random suffix if the generated slug already exists.

## Implementation Notes

- Add `slug VARCHAR(100) UNIQUE NOT NULL` to `organizations` table in `init.sql`.
- Generate slug from organization name during signup: lowercase, replace spaces/special chars with hyphens, truncate to 100 chars.
- Add unique index on `slug`.
