# ADR-004: Integration Tests with Real Database

## Status

Accepted

## Date

2026-04-06

## Context

The existing backend test (health.test.ts) uses Jest with mocked pg-promise. The ticket type feature introduces CRUD operations with SQL queries, uniqueness constraints, and foreign key checks. The team needed to decide whether to continue with mocked DB tests or use a real database for testing.

## Decision

Use integration tests with a real PostgreSQL database for the ticket type feature. Tests will run against the same Docker-managed PostgreSQL instance (or a dedicated test database) to validate SQL queries, constraints, and data integrity end-to-end.

## Alternatives Considered

### Alternative 1: Unit Tests with Mocked Database

- **Description**: Continue the existing pattern — mock pg-promise and test route handlers with Supertest.
- **Pros**: Fast execution, no infrastructure dependency, consistent with existing tests.
- **Cons**: Mocks can diverge from real database behavior; SQL syntax errors, constraint violations, and query logic bugs go undetected until runtime.
- **Why rejected**: The ticket type feature relies heavily on database constraints (unique names, foreign keys, cascade behavior). Mocking these would give false confidence.

### Alternative 2: Both Unit and Integration Tests

- **Description**: Unit tests for business logic + integration tests for database-touching paths.
- **Pros**: Comprehensive coverage at multiple levels.
- **Cons**: Doubles testing effort and maintenance burden for a feature with minimal pure business logic (most logic is CRUD + validation).
- **Why rejected**: The ticket type service is thin — nearly all logic involves database interaction. Separate unit tests would mostly duplicate integration test coverage without adding value.

## Consequences

### Positive

- Tests validate real SQL queries against actual PostgreSQL
- Database constraints (unique, FK) are tested as they behave in production
- Higher confidence in data integrity and query correctness
- Catches issues that mocks would hide (SQL syntax, type mismatches)

### Negative

- Tests are slower than mocked tests (database I/O)
- Requires a running PostgreSQL instance during test execution
- Test setup/teardown must manage database state (transactions or truncation)

### Risks

- Flaky tests due to shared database state. Mitigation: use transaction rollback or table truncation between tests.
- CI pipeline needs a PostgreSQL service. Mitigation: Docker Compose already provides PostgreSQL; CI can use the same setup.

## Implementation Notes

Create a test helper that connects to the database before tests and provides cleanup between test cases (truncate tables or wrap each test in a rolled-back transaction). Backend test command should ensure the database is running before executing.
