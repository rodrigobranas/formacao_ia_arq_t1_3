# ADR-003: React Router v7 for Frontend Navigation

## Status

Accepted

## Date

2026-04-06

## Context

The frontend is a single-component React app with no routing. The ticket type management feature requires a Settings area with sidebar navigation and a content area, implying multiple views and URL-based navigation. The team needed to choose a routing solution.

## Decision

Use React Router v7 for client-side routing. It will provide nested route layouts (Settings layout with sidebar + outlet for content) and URL-driven navigation between application areas.

## Alternatives Considered

### Alternative 1: TanStack Router

- **Description**: Type-safe routing library with built-in search param validation and file-based route generation.
- **Pros**: Strong TypeScript integration, modern API, type-safe route params.
- **Cons**: Steeper learning curve, smaller community, less beginner-friendly documentation.
- **Why rejected**: The project targets beginners. React Router is the industry standard with extensive learning resources and a gentler onboarding path.

### Alternative 2: State-Based View Switching

- **Description**: Use React state to toggle between views without a routing library. No URL changes.
- **Pros**: Zero dependencies, simplest possible implementation.
- **Cons**: No URL support (can't bookmark or share links to settings), no browser back/forward navigation, doesn't scale.
- **Why rejected**: A Settings area implies navigable URLs. Losing URL support degrades UX and makes the app feel non-standard.

## Consequences

### Positive

- Industry-standard routing with extensive documentation
- Nested layouts enable the Settings sidebar pattern naturally
- URL-driven navigation supports bookmarking and browser history
- Large ecosystem of examples and community support

### Negative

- Adds a dependency (~15 KB gzipped) to the frontend bundle
- Requires restructuring `App.tsx` to use router provider and route definitions

### Risks

- React Router v7 introduced breaking changes from v6. Mitigation: use the latest stable v7 API from the start; no migration needed.

## Implementation Notes

Define a root layout route, a Settings layout route (with sidebar), and a TicketTypes page route. Use `<Outlet />` for nested content rendering. The Settings sidebar renders navigation links using `<NavLink>` for active state styling.
