# ADR-002: Layered Backend Organization with One File Per Concern

## Status

Accepted

## Date

2026-04-06

## Context

The backend currently has all logic in a single `index.ts` file (Express app setup + health endpoint). The ticket type feature introduces CRUD operations with validation and database queries. The team needed to decide how to organize this new backend code while keeping the project beginner-friendly.

## Decision

Organize backend code as one file per concern in a flat `src/` directory: route files (e.g., `ticketTypeRoutes.ts`) and service/data-access files (e.g., `ticketTypeService.ts`). Routes handle HTTP request/response; services handle business logic and database queries. No nested feature folders.

## Alternatives Considered

### Alternative 1: Feature Folder Structure

- **Description**: A `ticket-types/` folder containing route, service, and model files together.
- **Pros**: Groups related files, scales well for many features.
- **Cons**: Adds directory nesting that may confuse beginners; premature for a project with only one domain entity.
- **Why rejected**: The project is an educational skeleton targeting beginners. Flat structure with clear file names is easier to navigate and understand.

### Alternative 2: Everything in index.ts

- **Description**: Add ticket type routes and logic directly in `index.ts` with helper functions in a single additional file.
- **Pros**: Minimal file count, simple to find code.
- **Cons**: `index.ts` grows quickly; mixes concerns; harder to test individual layers.
- **Why rejected**: Even for beginners, separating routes from data access teaches good practices and makes testing easier.

## Consequences

### Positive

- Clear separation of HTTP handling from business logic
- Easy to locate files — flat directory, descriptive names
- Consistent with beginner-friendly project philosophy
- Each file is independently testable

### Negative

- As features grow, flat `src/` may become crowded (acceptable for Phase 1 scope)

### Risks

- If many features are added later, the flat structure may need to evolve into feature folders. Mitigation: current scope is small; revisit when a third domain entity is introduced.

## Implementation Notes

File naming convention: `<entityName>Routes.ts` for route handlers, `<entityName>Service.ts` for business logic and data access. The `index.ts` file imports and mounts route modules using `app.use()`.
