# ADR-005: Forward-Declare Tickets Table for Foreign Key Delete Protection

## Status

Accepted

## Date

2026-04-06

## Context

The PRD requires that ticket types in use by existing tickets cannot be deleted. However, the tickets table and CRUD operations are out of scope for Phase 1 (ticket type management). The team needed to decide how to implement the delete protection constraint before the tickets feature exists.

## Decision

Create a minimal `tickets` table in `init.sql` with a foreign key referencing `ticket_types(id)`. This table will only contain the columns needed for the FK relationship (id, ticket_type_id). The delete endpoint will query this table to check for referencing rows before allowing deletion. This ensures the constraint and protection logic exist from day one.

## Alternatives Considered

### Alternative 1: Soft Check Without Foreign Key

- **Description**: The delete endpoint queries for referencing rows but no FK constraint exists in the database. Add the FK when the tickets feature is implemented.
- **Pros**: No need to create a table that isn't fully used yet.
- **Cons**: Database doesn't enforce referential integrity; application-level checks can be bypassed by direct SQL.
- **Why rejected**: Deferring the FK constraint means the database allows orphaned tickets if data is inserted by any means other than the application. Database-level integrity is more reliable.

### Alternative 2: Skip Delete Protection in Phase 1

- **Description**: Allow all deletions in Phase 1. Add protection when tickets are implemented.
- **Pros**: Fastest to implement, no forward declarations.
- **Cons**: Violates a PRD requirement. If tickets are created via seed data or manual SQL before Phase 2, orphaned references can occur.
- **Why rejected**: Delete protection is an explicit PRD requirement and a data integrity guarantee. Skipping it creates risk.

## Consequences

### Positive

- Referential integrity enforced at the database level from day one
- Delete protection logic is tested and working before tickets feature arrives
- Phase 2 (tickets CRUD) can build on an existing, proven constraint

### Negative

- A partially-defined `tickets` table exists before the tickets feature is implemented
- The table schema will need to be extended when tickets CRUD is built

### Risks

- The forward-declared tickets table may diverge from what Phase 2 actually needs. Mitigation: keep the table minimal (only id + ticket_type_id); Phase 2 adds columns via schema extension.

## Implementation Notes

The `tickets` table in `init.sql` should contain only: `id SERIAL PRIMARY KEY`, `ticket_type_id INTEGER NOT NULL REFERENCES ticket_types(id)`. No seed data for tickets. The delete endpoint checks `SELECT COUNT(*) FROM tickets WHERE ticket_type_id = $1` before proceeding.
