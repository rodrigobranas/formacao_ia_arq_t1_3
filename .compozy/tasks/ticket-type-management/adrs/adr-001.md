# ADR-001: Inline Settings Page for Ticket Type Management

## Status

Accepted

## Date

2026-04-06

## Context

The ticket management system needs a way for users to create, read, update, and delete ticket types (e.g., Dúvida, Inconsistência, Sugestão, Customização). These types are a foundational entity used when opening tickets. The team needed to decide on the UI pattern for managing this configuration entity within a Settings area.

## Decision

Use an inline settings page approach: a single Settings > Ticket Types page with a table listing all types. Users add and edit types directly via inline form fields within the table. This keeps the interaction fast, reduces navigation overhead, and is well-suited for a simple entity with only two fields (name and description).

## Alternatives Considered

### Alternative 1: Dedicated CRUD with Detail Page

- **Description**: Separate list and form pages with full-page navigation for create/edit operations.
- **Pros**: Structured approach, scales well for entities with many fields, supports rich detail pages.
- **Cons**: Excessive navigation overhead for a two-field entity, more pages to build and maintain.
- **Why rejected**: Ticket types are simple entities (name + description). Full-page navigation adds unnecessary complexity without proportional benefit.

### Alternative 2: Modal-based CRUD

- **Description**: Table view with modal dialog overlays for add/edit operations.
- **Pros**: Good balance between simplicity and separation of concerns, no page navigation needed.
- **Cons**: Modals can feel heavy for a two-field form, accessibility concerns with overlays, less inline feel.
- **Why rejected**: While viable, inline editing is more direct and lightweight for such a simple entity. Modals would be more appropriate for forms with more fields or complex validation.

## Consequences

### Positive

- Minimal navigation — users stay on one page for all CRUD operations
- Fast interaction — inline editing reduces clicks and page loads
- Simple to implement — fewer routes and components needed
- Extensible — the Settings sidebar can accommodate future configuration entities

### Negative

- Inline editing can be harder to implement with complex validation
- Limited screen real estate if many fields are added to ticket types in the future

### Risks

- If ticket types grow in complexity (more fields), the inline pattern may need to evolve into a modal or separate form. Mitigation: the current scope is deliberately limited to name + description.

## Implementation Notes

The Settings page should include a sidebar navigation to accommodate future settings categories. The Ticket Types section should be the first entry.
