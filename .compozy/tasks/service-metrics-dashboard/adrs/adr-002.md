# ADR-002: Charting Library — Recharts

## Status

Accepted

## Date

2026-04-07

## Context

The service metrics dashboard requires a charting library for rendering line charts, bar charts, and heatmaps. The frontend stack is React 18 + Vite + Tailwind CSS + shadcn/ui. No charting library is currently installed. The library must support TypeScript, integrate well with React's component model, and handle the chart types defined in the PRD (line, bar, stacked bar, horizontal bar, heatmap).

## Decision

Use **Recharts** as the charting library. It is built on React and D3, provides declarative composable components, has strong TypeScript support, and is the most widely adopted React charting library.

## Alternatives Considered

### Alternative 1: Chart.js + react-chartjs-2

- **Description**: Canvas-based charting library with a React wrapper.
- **Pros**: Very performant for large datasets; battle-tested; well-documented.
- **Cons**: Canvas-based rendering is less React-idiomatic; harder to style with Tailwind; imperative API underneath.
- **Why rejected**: The dataset size for this dashboard is small (aggregated metrics, not raw data). The React-idiomatic composability of Recharts is more valuable than Chart.js's canvas performance.

### Alternative 2: Tremor

- **Description**: Dashboard component library built on Tailwind + Recharts, providing pre-built KPI cards and chart components.
- **Pros**: Higher-level abstraction; pre-built dashboard components; Tailwind-native.
- **Cons**: Adds a heavy dependency for components we can build with shadcn/ui + Recharts; less control over styling; larger bundle.
- **Why rejected**: The project already uses shadcn/ui for UI components. Adding Tremor would introduce overlapping abstractions. Building KPI cards with shadcn/ui + Recharts gives more control.

## Consequences

### Positive

- Declarative, composable API aligns with React patterns used throughout the frontend
- Strong TypeScript support reduces integration friction
- Lightweight — only adds the charts we need without bundling a full dashboard framework

### Negative

- Heatmap is not a built-in Recharts chart type — will need a custom implementation or a simple grid-based approach
- SVG-based rendering could be slower than canvas for very large datasets (not a concern at current scale)

### Risks

- If data volume grows significantly in future phases, SVG rendering may need optimization — mitigated by the small aggregated dataset size for dashboard metrics

## Implementation Notes

Install via `npm install recharts` in the frontend package.
