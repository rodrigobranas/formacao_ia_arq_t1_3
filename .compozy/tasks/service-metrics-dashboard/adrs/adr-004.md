# ADR-004: API Design — Single Metrics Endpoint with Polling Refresh

## Status

Accepted

## Date

2026-04-07

## Context

The dashboard frontend needs to fetch KPIs, queue summary, and trend data from the backend. The real-time view auto-refreshes every 30 seconds. The decision involves both the API structure (single vs. multiple endpoints) and the frontend refresh mechanism.

## Decision

Use a **single endpoint** `GET /api/dashboard/metrics` that returns all dashboard data (KPIs, queue summary, and trend data) in one response. The frontend refreshes using a simple **`setInterval` polling every 30 seconds** for the real-time view. Trend data is fetched on tab switch with a `period` query parameter.

## Alternatives Considered

### Alternative 1: Two Endpoints (Real-time + Trends)

- **Description**: `GET /api/dashboard/realtime` for KPIs and queue, `GET /api/dashboard/trends?period=7d` for charts. Independent refresh rates.
- **Pros**: Allows refreshing real-time data frequently without re-fetching trends; smaller payloads per request.
- **Cons**: Two network calls on initial load; frontend must coordinate two data sources; adds API surface.
- **Why rejected**: The combined payload is small (KPIs are numbers, queue is 10-15 items, trends are 90 data points max). A single request is simpler and eliminates coordination complexity.

### Alternative 2: Multiple Granular Endpoints

- **Description**: Separate endpoints per widget: `/api/dashboard/kpis`, `/api/dashboard/queue`, `/api/dashboard/volume-trend`, etc.
- **Pros**: Maximum flexibility; each widget can refresh independently; easiest to cache individually.
- **Cons**: 5+ network calls per page load; complex orchestration; harder error handling; more API surface to maintain.
- **Why rejected**: Over-engineered for the current requirements. The dashboard is a single view for a single persona.

### Alternative 3: React Query with refetchInterval

- **Description**: Use TanStack React Query for caching, deduplication, background refetching, and stale-while-revalidate.
- **Pros**: Better data lifecycle management; built-in loading/error states; deduplication.
- **Cons**: Adds a dependency; team is not using it elsewhere; overkill for a single polling endpoint.
- **Why rejected**: The codebase uses plain `fetch` throughout. Adding React Query for one page would be inconsistent. A simple `setInterval` + `useState` matches existing patterns.

## Consequences

### Positive

- Single network call per refresh — simple, predictable, easy to monitor
- Consistent with existing codebase patterns (plain fetch + useState)
- No new frontend dependencies for data fetching

### Negative

- Trend data is re-fetched on every 30s poll even when the user is on the real-time tab — mitigated by making trend data optional via query parameter
- No automatic deduplication or caching — if the user navigates away and back, data is re-fetched from scratch

### Risks

- If the single response payload grows too large in future phases, may need to split — mitigated by the `period` query parameter that controls trend data inclusion

## Implementation Notes

- `GET /api/dashboard/metrics` — returns KPIs + queue (always)
- `GET /api/dashboard/metrics?period=7d` — returns KPIs + queue + trend data for the given period
- Frontend polls every 30s without `period` param for the real-time tab; includes `period` param when on the trends tab
