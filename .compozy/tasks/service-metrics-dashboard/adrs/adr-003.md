# ADR-003: Metrics Data Strategy — On-the-fly SQL Aggregation

## Status

Accepted

## Date

2026-04-07

## Context

The dashboard needs to compute KPIs (open count, unassigned count, avg resolution time, etc.) and historical trend data (ticket volume per day, resolution time per day). The existing database is PostgreSQL with pg-promise. No aggregation infrastructure exists. The decision is whether to compute metrics in real-time from the tickets table or pre-compute them into dedicated metrics tables.

## Decision

Use **on-the-fly SQL aggregation queries** against the existing `tickets` table. KPIs and trend data are computed directly using `COUNT`, `AVG`, `GROUP BY`, and date functions at query time. No additional tables or scheduled jobs are needed.

## Alternatives Considered

### Alternative 1: Pre-computed Metrics Table

- **Description**: A scheduled job (cron or pg_cron) periodically aggregates ticket data into a `dashboard_metrics` table. The API reads from this table.
- **Pros**: Faster reads; constant query time regardless of ticket volume; offloads computation from request path.
- **Cons**: Adds schema complexity; data staleness window between job runs; requires job scheduling infrastructure; harder to debug discrepancies.
- **Why rejected**: Current ticket volume does not justify the added complexity. Pre-computation is premature optimization.

### Alternative 2: Hybrid (Live KPIs + Pre-computed Trends)

- **Description**: Real-time KPIs via live queries; historical trends via a `daily_stats` table populated nightly.
- **Pros**: Best of both worlds — live KPIs with fast trend queries.
- **Cons**: Still requires a scheduled job and extra table; adds two data paths to maintain.
- **Why rejected**: Unnecessary complexity for MVP. Trend queries over 90 days on the tickets table are fast enough with proper indexing.

## Consequences

### Positive

- Zero additional schema or infrastructure — uses existing tables only
- Data is always current — no staleness window
- Simpler to debug — one source of truth

### Negative

- Query cost grows linearly with ticket count — may need optimization at scale
- Multiple aggregation queries per request add latency vs. a single table read

### Risks

- If ticket volume exceeds tens of thousands of rows, aggregation queries may slow down — mitigated by adding database indexes on `status`, `created_at`, and `organization_id` columns

## Implementation Notes

Add composite indexes on the `tickets` table to support efficient aggregation:
- `(organization_id, status)` — for KPI counts
- `(organization_id, created_at)` — for trend grouping
