# ADR-001: Flat Monorepo Structure (frontend/ + backend/)

## Status

Accepted

## Date

2026-04-06

## Context

We need to define the directory structure for a fullstack monorepo that serves as a base architecture for an educational course targeting beginner developers. The project uses npm workspaces with React+Vite+Tailwind+shadcn on the frontend and Node+Express+PostgreSQL+pg-promise on the backend. The structure must be easy to understand and navigate for students with limited experience.

## Decision

Use a flat monorepo structure with two top-level workspace directories: `frontend/` and `backend/`. Each is a self-contained npm workspace. No shared packages or additional abstraction layers.

## Alternatives Considered

### Alternative 1: apps/ + packages/shared

- **Description**: Standard monorepo layout with `apps/frontend`, `apps/backend`, and `packages/shared` for shared TypeScript types and contracts.
- **Pros**: Industry-standard pattern, enables shared types without duplication, teaches workspace dependency management.
- **Cons**: Adds a third package to maintain, introduces workspace dependency concepts that may overwhelm beginners, more indirection in project navigation.
- **Why rejected**: Too much structural complexity for a beginner-focused course. The shared package concept can be introduced later as students advance.

### Alternative 2: apps/ + packages/shared + packages/ui

- **Description**: Extended layout adding a shared UI component library alongside shared types.
- **Pros**: Teaches library publishing patterns, mirrors large-scale enterprise monorepos.
- **Cons**: Significantly more complex, requires understanding of library packaging, overkill for a beginner course.
- **Why rejected**: Appropriate for advanced courses, not for beginners. Adds complexity without proportional learning value at this level.

## Consequences

### Positive

- Students can immediately understand the project layout — two clearly named folders
- Reduced cognitive load allows focus on learning React, Express, and database patterns
- Each workspace is fully self-contained and can be understood independently

### Negative

- No shared type package means TypeScript types for API contracts may need to be duplicated between frontend and backend
- Does not teach workspace dependency patterns (importing from sibling packages)

### Risks

- Students may develop habits of duplicating types; mitigated by introducing shared packages in a follow-up course module

## Implementation Notes

Root `package.json` will declare workspaces as `["frontend", "backend"]`. Each workspace has its own `package.json`, `tsconfig.json`, and source directory.
