# ADR-003: Default Organization Migration Strategy for Tenant Scoping

## Status

Accepted

## Date

2026-04-07

## Context

The existing database has `ticket_types` and `tickets` tables without any organization scoping. Introducing multi-tenancy requires adding an `organization_id` foreign key to these tables. A migration strategy is needed for existing data.

## Decision

Create a "Default" organization during migration and assign all existing `ticket_types` and `tickets` rows to it. The `organization_id` column is added as NOT NULL with a foreign key constraint, ensuring tenant isolation is enforced at the database level from day one.

## Alternatives Considered

### Alternative 1: Wipe Existing Data

- **Description**: Drop all existing ticket_types and tickets data during migration.
- **Pros**: Cleanest approach, no legacy data concerns.
- **Cons**: Destructive, loses any production data.
- **Why rejected**: Unnecessarily destructive even if current data is seed data — the default org approach is equally simple and non-destructive.

### Alternative 2: Nullable organization_id

- **Description**: Add organization_id as nullable, backfill gradually.
- **Pros**: Non-breaking migration, no immediate data changes needed.
- **Cons**: Queries must handle NULL org, weakens tenant isolation guarantees, adds complexity to every query.
- **Why rejected**: Defeats the purpose of strict tenant isolation. NULL organization means data belongs to no tenant, creating ambiguity.

## Consequences

### Positive

- Zero data loss during migration
- NOT NULL constraint enforced immediately — no orphaned rows
- Clean, simple migration script
- Default organization can be used for development and testing

### Negative

- A "Default" organization exists that may not correspond to a real customer
- Existing data semantics change (rows now "belong" to the default org)

### Risks

- Default organization could accumulate test data — mitigated by documenting it as a development artifact

## Implementation Notes

- Migration creates the default organization first, then adds `organization_id` column with DEFAULT pointing to it, then removes the DEFAULT
- All existing queries for ticket_types and tickets must be updated to filter by organization_id
