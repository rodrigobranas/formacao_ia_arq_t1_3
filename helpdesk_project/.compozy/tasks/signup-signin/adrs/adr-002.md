# ADR-002: JWT Stateless Authentication with bcrypt Password Hashing

## Status

Accepted

## Date

2026-04-07

## Context

The application needs an authentication mechanism for user signin and session management. The backend is a stateless Express.js API serving a React SPA. The team needs to choose between server-side sessions and token-based authentication, as well as a password hashing algorithm.

## Decision

Use stateless JWT tokens for authentication. On successful signin, the backend issues a signed JWT containing the user ID, organization ID, and admin flag. The frontend stores the token in localStorage and sends it via the Authorization header on every request. Passwords are hashed using bcrypt before storage.

## Alternatives Considered

### Alternative 1: Server-side Sessions

- **Description**: Store sessions in PostgreSQL or Redis, issue session ID cookies.
- **Pros**: Easy server-side revocation, no token size concerns.
- **Cons**: Requires session store infrastructure, adds database queries on every request, stateful.
- **Why rejected**: Adds infrastructure complexity unnecessary for MVP. The app has no requirement for immediate session revocation.

### Alternative 2: Argon2 for Password Hashing

- **Description**: Use the argon2 algorithm instead of bcrypt.
- **Pros**: Newer, more resistant to GPU attacks, winner of Password Hashing Competition.
- **Cons**: More complex setup, less ecosystem support in Node.js.
- **Why rejected**: bcrypt is industry-standard, well-tested, and sufficient for MVP security requirements. Migration to argon2 remains possible in the future.

## Consequences

### Positive

- No server-side session state to manage
- Stateless backend scales horizontally without session synchronization
- bcrypt is battle-tested with simple API (auto-salt, configurable rounds)
- JWT payload carries user context, reducing database lookups on authenticated requests

### Negative

- Tokens cannot be revoked server-side without additional infrastructure (blacklist)
- localStorage is vulnerable to XSS (acceptable trade-off for MVP)
- JWT tokens are larger than session IDs

### Risks

- XSS vulnerability could expose JWT token — mitigated by sanitizing user inputs and using Content Security Policy headers in future phases
- Token expiration must be balanced between security and UX — start with 24-hour expiry

## Implementation Notes

- Use the `jsonwebtoken` npm package for JWT signing and verification
- Use `bcrypt` npm package for password hashing with 10 salt rounds
- JWT secret stored in environment variable (`JWT_SECRET`)
- Token payload: `{ userId, organizationId, admin }`
- Auth middleware extracts and verifies token from `Authorization: Bearer <token>` header
