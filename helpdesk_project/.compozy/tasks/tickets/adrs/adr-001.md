# ADR-001: Ticket Status Workflow — Lean Linear Flow

## Status

Accepted

## Date

2026-04-07

## Context

The ticket management system needs a status workflow to track the lifecycle of support tickets from creation to resolution. The workflow must support public ticket submission without login, operator assignment, forwarding between operators, and ticket closure. The primary users are end customers submitting tickets and logged-in operators managing them.

## Decision

Adopt a Lean Linear Flow with three statuses: **New → Assigned → Closed**.

- Tickets are created with status `New` by end customers via a public form (no login required).
- An operator claims the ticket by clicking "Assign to me", changing status to `Assigned`. The assignment is logged in a separate history table.
- Forwarding a ticket to another operator keeps status as `Assigned` and logs the new assignee in the history table.
- The operator adds comments (with optional file attachments) and closes the ticket, changing status to `Closed`.
- Customers can track their ticket status via a unique ticket code without logging in.

## Alternatives Considered

### Alternative 1: Standard Helpdesk Flow (5 statuses)

- **Description**: New → Assigned → In Progress → Resolved → Closed
- **Pros**: More granular visibility into ticket progress; Resolved status allows customer confirmation before closing
- **Cons**: More clicks per ticket lifecycle; additional complexity for operators; more states to manage in the UI
- **Why rejected**: Overengineered for the MVP requirements. The user explicitly requested a simple system. Granular tracking can be added in a future phase if needed.

### Alternative 2: Queue-Based Flow (4 statuses)

- **Description**: New → In Progress → Resolved → Closed. Assigning auto-moves to In Progress; forwarding returns to New.
- **Pros**: Assignment equals action, reducing ambiguity about whether work has started
- **Cons**: Forwarded tickets lose their "in progress" state and return to queue; semantically confusing when a ticket bounces between New and In Progress
- **Why rejected**: The state reset on forwarding creates a misleading history. The lean approach with explicit assignment logging provides clearer traceability without the confusion.

## Consequences

### Positive

- Minimal cognitive overhead for operators — only three states to manage
- Fast to implement — fewer transitions, simpler UI, less validation logic
- Clear traceability via the assignment history table despite few statuses

### Negative

- No visibility into whether an assigned ticket is actively being worked on
- No "Resolved" intermediate state — tickets go directly from Assigned to Closed

### Risks

- If operators need to distinguish "assigned but not started" from "actively working", the workflow may need a fourth status later. Mitigation: the assignment history table provides some proxy for activity tracking.

## Implementation Notes

Status transitions allowed: New → Assigned, Assigned → Closed. No backward transitions in MVP.
